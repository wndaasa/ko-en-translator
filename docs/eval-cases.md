# 실측 평가 케이스 (ozai 협업 에디터 연동, 2026-07-03)

ozai_nb2 협업 에디터의 번역 기능(버블 메뉴 `기능>번역`)에 서비스로 탑재하면서 나온
**실사용 실패/성공 사례 모음**. 다음 학습 라운드의 전/후 비교 벤치마크로 쓴다 —
BLEU와 별개로, 실제 사용자가 넣는 입력에서의 체감 품질을 고정 케이스로 추적한다.

- 측정 대상: NPU 서버(CPU) 서빙, sentence-base + doc-paper 2모델 자동 라우팅
- 서빙 구성(당시): 토큰 길이 ≥70 & to=en → doc-paper 문맥모드 / 그 외 → sentence-base
  문장 분리. to=ko는 길이 무관 sentence-base(doc-paper는 `--no-bidi` ko→en 단방향).
  550자 초과 to=en은 sentence 폴백(doc 디코딩 O(L²) 타임아웃 회피, 응답 note 표기).
- 후처리(서빙): 자막 대사마커·반복 제거(clean_subtitle), en→ko 원문 에코 제거
  (타깃 반대 문자 4단어+ 연속 제거, GPU/minRNN 등 짧은 약어 보존)

## 케이스 1 — 자막 아티팩트 (해결됨, 회귀 감지용)

- 입력(en→ko): `Hello.`
- 수정 전: ` - 안녕하세요 - 안녕하세요` — OPUS 자막 코퍼스의 대화 대시 라인 패턴 누출.
  도메인 태그(casual/formal)로도 교정 불가 확인.
- 수정 후: `안녕하세요` ✅ (서빙 후처리로 해결. **근본 해결은 자막 노이즈 정제/재학습**)

## 케이스 2 — 문단 ko→en: 라우팅 前/後 (서빙 해결됨)

입력(302자 5문장):

> 이 프로젝트는 한국어와 영어 사이의 번역을 위해 트랜스포머를 바닥부터 구현하는 것으로 시작했습니다. 이후 어텐션을 minGRU 기반의 선형 순환으로 대체해 메모리 효율을 크게 개선했고, 대규모 병렬 코퍼스로 학습을 진행했습니다. 특히 논문 도메인에 특화된 파인튜닝을 통해 문서 단위 번역에서 BLEU 35 이상의 성능을 달성했습니다. 현재는 협업 문서 편집기에 번역 기능으로 탑재되어, 사용자가 선택한 텍스트를 실시간으로 번역하고 있습니다. 앞으로는 회사의 GPU 자원을 활용해 더 큰 모델을 학습하고, 도메인 커버리지를 넓혀갈 계획입니다.

- sentence-base에 통짜 입력(수정 전): 첫 문장 유실 + `which is currently being used
  in...` 반복 붕괴 — 문장 단위 학습 모델의 분포 밖 입력 증상.
- doc-paper 문맥모드(수정 후): 5문장 모두 보존, 반복 없음 ✅ (~9s → 재측정 4s)
- 남은 모델 과제(오역): `바닥부터` → *from floor to*, `코퍼스` → *copper*(!),
  `달성했습니다` → *it is a matter of achieving* 류의 다의어/용어 오역.

## 케이스 3 — en→ko 에코 (서빙 해결됨)

- 증상: 일부 문장이 "한국어 번역문 + 영어 원문"으로 중복 출력
  (`...서비스로 전송됩니다.When a user selects text and clicks...`).
- 원인: 원문 fallback 로직이 아니라 **모델 에코**(원문을 변형 재생성해 verbatim 매칭 불가).
- 해결: 스크립트 기준(타깃 언어 반대 문자 4단어+ 연속) 제거 후처리 ✅
- 근본 과제: en→ko 방향 자체가 약함(문맥 파인튜닝이 ko→en 단방향이었음).

## 케이스 4 — 실전 학술 초록 en→ko (★모델 과제 집합, 최우선 벤치마크)

입력은 심사 중 논문의 초록(781자, en→ko)이라 **원문/출력 전문은 비공개**
(`docs/eval-cases.private.md`, git 제외 — 게재 확정 후 공개본 이관). 정답 번역을
직접 작성할 수 있는 실전 케이스라 전/후 비교의 기준점으로 쓴다.

오류 분류(전부 모델 수준 — 학습으로만 해결):

| 오류 | 양상 | 원인 분류 |
|---|---|---|
| 다의어 오역 | (AI) generation→"세대", prompts→"초기의", X-based→"X에 본사를 둔" | 뉴스/자막 도메인 편향 |
| 고유명사 훼손/유실 | 미학습 시스템 약어가 다른 철자로 변형되거나 문장에서 통째로 증발 | 서브워드 재조립 실패 |
| 부정 스코프 유실 | "rarely A or B"에서 부정이 B에 미적용(의미 반전) | 복문 구조 처리 한계 |
| 절 유실 | 동격 명사구를 동반한 긴 절이 통째로 소실 | 긴 복문 + 용량 한계(268M) |

## 케이스 5 — 성능/동시성 실측

| 항목 | 수치 |
|---|---|
| sentence 문장당 | ~0.2–0.3s (CPU) |
| doc 문맥모드 | 208자 4.7s / 302자 9s → O(L²) (스텝마다 전체 prefix 재디코딩) |
| 550자 초과 폴백(sentence) | ~900자 6–7s |
| 동시 12발 | 200×8(계단식 직렬) + 429×4(대기열 상한), 서비스 생존 |

## 개선 로드맵 후보 (우선순위 제안)

1. **beam search(4~5)** — 재학습 불필요, 디코딩만 교체. 통상 BLEU +1~2, 반복 억제.
2. **doc 디코딩 O(L) 증분화** — `MinGRU.step()` 기반 증분 디코딩(현재 별도 이슈).
   완성 시 550자 폴백 자체가 불필요.
3. **학술/기술 병렬 데이터 보강 + 자막 노이즈 정제** — 케이스 2·4의 용어 오역 직격.
4. **EXAONE(33B, 동일 NPU) 증류** — 합성 병렬 데이터 대량 생성 → 소형 모델 학습.
5. **모델 사이즈 업**(268M → 0.5~1B) — 케이스 4의 복문/부정 처리.
6. **en→ko 문서모드** — 문맥 파인튜닝을 `--no-bidi` 없이(양방향) 재수행.
7. **용어 보존 후처리**(서빙) — 원문 약어/식별자 추출 → 출력의 훼손 변형 복원.

## 실사용 데이터 수집

ozai 서버가 번역 성공 건을 `server/storage/translation-log/YYYY-MM.jsonl`에 적재 중
(`TRANSLATION_LOGGING=true`, 필드: ts/userId/to/source/translation/ms). 수 개월 수집 후
중복 제거·언어 필터·실패 사례 분리를 거쳐 학습용 TSV로 변환해 재학습에 투입 예정.
