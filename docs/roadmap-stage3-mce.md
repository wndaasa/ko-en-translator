# Stage 3: MCE — 형태소 합성 인코더 (실행 완료)

> 구현·검증·비교 학습까지 수행함. 아래 "설계"는 합의된 스펙, "실행 결과"는 측정값과 결론.
> **요약 결론: 음성 결과** — 동일 step에서 MCE가 BPE를 일관되게 밑돌고, 학습은 ~5배 느림.
> 가설(형태소 합성이 저빈도 용어를 개선)은 본 설정에서 지지되지 않았다. BPE가 강한 베이스라인.

## 동기

Stage 2에서 본 오역은 대부분 **저빈도 전문어**(`말뭉치`→"distal drought", `벤치마크`→"ventmark")에서
발생했다. 원인은 플랫 BPE가 드문 단어를 예측 불가능하게 분절하기 때문이다. 한국어는 교착어
(`먹+었+습니다`)라 단어 내부에 형태소 구조가 있는데 플랫 BPE는 이를 활용하지 못한다.

**가설**: 단어를 문자(한글=음절)에서 *합성*해 인코더에 넣으면, 처음 보는 단어도 구성 요소로부터
의미를 만들 수 있어 저빈도 용어 일반화가 개선된다.

## 설계 (MCE: Morpheme-Compositional Encoder)

플랫 BPE 토큰 임베딩 대신, 소스측 인코더 앞단을 **문자→단어 계층 합성**으로 교체한다.

```
소스 문장
  └─ 공백으로 단어 분리 (한·영 공통)
  [L1] 문자 임베딩      한글=음절, 영어=알파벳, 숫자/기호=각 문자 (소스 전용 문자 vocab ~3-5k)
  [L2] CharCNN 합성기   단어의 문자열 → 단어 벡터 1개
                        (conv 폭 [1,2,3,4,5] → max-pool → highway → d_model 투영)
  [L3] 단어 트랜스포머   기존 인코더 N층 (단어 단위) + 위치인코딩 + 방향태그 벡터
  → 디코더 cross-attention (디코더·타깃 BPE 전부 베이스라인과 동일)
```

**확정된 선택**
- L1 단위: **음절**(한글 블록). 자모는 향후 ablation 후보.
- L2 합성기: **CharCNN** (빠르고 검증된 형태소 n-gram 포착, ROCm 친화적).
- 적용 범위: **한·영 공통 문자합성** (한국어 음절·영어 알파벳에 같은 합성기). 양방향 일관.
- 방향 태그(`<2en>/<2ko>`)는 문자합성 우회 → L3 단어열 맨 앞 학습형 벡터.

**파라미터 매칭**: 베이스라인 인코더측 32k×512 토큰 임베딩이 문자임베딩(~0.5M)+CharCNN(~1-2M)으로
대체되어 **총 파라미터는 비슷하거나 약간 적음**. "모델을 키워서 이긴 것"이 아님을 보장.

## 구현 단계 (베이스라인 무손상)

1. `src/char_tokenizer.py` — 소스 문자 vocab 구축/인코딩(단어→문자ids + 경계)
2. `src/char_encoder.py` — CharCNN 합성기 + MCE 인코더, 단위 테스트(shape/마스킹)
3. `src/model.py` — `encoder_type="bpe"|"mce"` 플래그로 MCE 추가(기존 경로 보존)
4. 데이터/collate — 소스(단어×문자) + 타깃(BPE) 배치
5. 토이 overfit으로 학습 동작 검증
6. OPUS 본 학습 → 베이스라인과 동일 데이터·스텝·평가로 BLEU 정면 비교
7. **전문어 테스트셋**(말뭉치/벤치마크류 50~100개) 별도 측정 → 가설 검증, (선택) 학술 파인튜닝 비교

## 평가 (성공/실패 모두 결과물)

- **H1 (핵심)**: 저빈도 전문어 번역 정확도 — 전용 테스트셋 + 전체 BLEU
- **H2**: 인코더 시퀀스 길이/속도 (단어 단위라 BPE보다 짧을 것으로 기대)
- 비교는 베이스라인 대비 ablation. MCE가 BPE를 못 이겨도, "왜 BPE가 강한가"의 분석이 산출물.

## 리스크

- 문헌상 문자/형태소 인코더가 BPE를 못 이기는 경우가 흔함 — 결과를 장담하지 않는다.
- 단어 합성기가 단어마다 동작 → 학습이 다소 느려질 수 있음.
- 한국어 공백(어절)이 형태소 경계와 완전히 일치하지는 않음(작동에는 무방).
- 순수 PyTorch 구현이라 ROCm 커널 리스크는 없음.

## 실행 결과

검증 단계는 모두 통과: 단위 테스트(shape/causal/패딩/weight tying), 토이 overfit(ko→en BLEU **100.0**,
구조 정상). 본 비교 학습은 베이스라인과 **동일 조건**(OPUS 양방향, batch 96, d_model 512, 6+6층,
lr 5e-4, warmup 4000)으로 수행. MCE 파라미터 **61.9M** (베이스라인 60.5M과 매칭 → "키워서 이긴 것" 아님).

### 트러블슈팅: MIOpen conv 재탐색으로 인한 학습 정체
- **증상**: 모델 빌드 후 16분간 step 로그 0. 메인 프로세스 CPU 389%, GPU·워커 유휴.
- **원인**: CharCNN의 conv 입력 배치 크기 `B×W`(W=문장당 단어 수)가 배치마다 크게 변함 →
  ROCm **MIOpen이 매 배치 conv 커널을 재탐색**(기본 FIND 모드). 토이(짧은 문장, W 거의 일정)에선
  안 나타났던 문제.
- **해결**: `MIOPEN_FIND_MODE=2`(휴리스틱 즉시 모드, 탐색 생략) → 즉시 정상 진행(225 ex/s).

### 동일-step 비교 (핵심, 공정)

학습 budget을 통제하기 위해 **같은 step에서의 검증 지표**를 비교한다.

| step | val_loss (base / MCE) | ko→en BLEU (base / MCE) |
|---|---|---|
| 2,000  | 5.83 / 5.92 | 0.45 / 0.14 |
| 6,000  | 5.08 / 5.34 | 1.88 / 0.28 |
| 10,000 | 4.73 / 5.12 | 2.62 / 0.62 |
| 14,000 | 4.55 / 4.99 | 2.58 / 0.96 |
| 18,000 | 4.40 / 4.88 | 4.54 / 0.89 |
| 22,000 | 4.31 / 4.81 | 4.16 / **1.51** |

- **11번의 eval 내내 MCE가 val_loss·BLEU 모두에서 일관되게 뒤짐.** 추세가 안정적이라
  더 학습해도 역전 가능성 낮음 → step 22,000(≈1.5 epoch)에서 비교 종료.
- **속도**: 베이스라인 ~1,150 ex/s vs MCE **~225 ex/s (약 5배 느림)**. 단어마다 CharCNN을
  돌리는 비용 + 가변 shape conv 오버헤드. → MCE는 per-step·per-wall-clock 둘 다 열세.

> 정성 스냅샷(참고용, MCE는 적은 step): MCE는 영어 유창성은 익혔으나 내용 매핑이 아직 약함
> (예: "저는 인공지능을 연구하고 있습니다" → base "I'm studying a E.R." / mce "I'm a surgeon.").
> step이 다르므로 정성 head-to-head는 공정하지 않음 — 결론은 위 동일-step 표에 근거.

### 결론과 해석

가설(문자 합성이 저빈도 용어 일반화를 도와 BLEU↑)은 **본 설정에서 지지되지 않음**. 추정 원인:
1. **BPE가 이미 강함**: 32k BPE가 이 코퍼스의 서브워드 구조를 충분히 포착. 음절 합성의 추가 이득이 작음.
2. **최적화 난이도**: CharCNN→단어 표현이 BPE 임베딩 룩업보다 학습이 느리고 어려움(같은 step에서 손해).
3. **어절 경계의 한계**: 한국어 공백은 형태소 경계와 불일치 → 합성 단위가 이상적이지 않음.

이는 흔한(그리고 정직한) 음성 결과다. "가설 → 공정한 설계 → 측정 → 분석"이 산출물이며, BPE가 강한
베이스라인임을 정량적으로 확인했다.

### 남은 과제 (했다면 더 볼 것)
- 저빈도 **전문어 전용 테스트셋**으로 H1을 직접 측정(이번엔 MCE가 per-step에서 이미 뒤져 우선순위 낮춤).
- 합성기 변형(char-Transformer), 자모 단위, 깊은 인코더, 서브워드 정규화(BPE-dropout) 등 ablation.
- 산출물: `runs/mce/best.pt`(step 22k, 비교용), 코드 `src/char_tokenizer.py`·`src/char_encoder.py`,
  `--arch mce` 학습 경로, 단위 테스트 `tests/test_mce.py`.
