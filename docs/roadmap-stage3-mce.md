# Stage 3 (계획): MCE — 형태소 합성 인코더

> 아직 구현 전. 이 문서는 합의된 설계 스펙과 실험 계획을 기록한다.

## 동기

Stage 2에서 본 오역은 대부분 **저빈도 전문어**(`말뭉치`→"distal drought", `벤치마크`→"ventmark")에서
발생했다. 원인은 플랫 BPE가 드문 단어를 예측 불가능하게 분절하기 때문이다. 한국어는 교착어
(`먹+었+습니다`)라 단어 내부에 형태소 구조가 있는데 플랫 BPE는 이를 활용하지 못한다.

**가설**: 단어를 문자(한글=음절)에서 *합성*해 인코더에 넣으면, 처음 보는 단어도 구성 요소로부터
의미를 만들 수 있어 저빈도 용어 일반화가 개선된다.

## 설계 (MCE: Morpheme-Compositional Encoder)

플랫 BPE 토큰 임베딩 대신, 소스측 인코더 앞단을 **문자→단어 계층 합성**으로 교체한다.

```
소스 문장
  └─ 공백으로 단어 분리 (한·영 공통)
  [L1] 문자 임베딩      한글=음절, 영어=알파벳, 숫자/기호=각 문자 (소스 전용 문자 vocab ~3-5k)
  [L2] CharCNN 합성기   단어의 문자열 → 단어 벡터 1개
                        (conv 폭 [1,2,3,4,5] → max-pool → highway → d_model 투영)
  [L3] 단어 트랜스포머   기존 인코더 N층 (단어 단위) + 위치인코딩 + 방향태그 벡터
  → 디코더 cross-attention (디코더·타깃 BPE 전부 베이스라인과 동일)
```

**확정된 선택**
- L1 단위: **음절**(한글 블록). 자모는 향후 ablation 후보.
- L2 합성기: **CharCNN** (빠르고 검증된 형태소 n-gram 포착, ROCm 친화적).
- 적용 범위: **한·영 공통 문자합성** (한국어 음절·영어 알파벳에 같은 합성기). 양방향 일관.
- 방향 태그(`<2en>/<2ko>`)는 문자합성 우회 → L3 단어열 맨 앞 학습형 벡터.

**파라미터 매칭**: 베이스라인 인코더측 32k×512 토큰 임베딩이 문자임베딩(~0.5M)+CharCNN(~1-2M)으로
대체되어 **총 파라미터는 비슷하거나 약간 적음**. "모델을 키워서 이긴 것"이 아님을 보장.

## 구현 단계 (베이스라인 무손상)

1. `src/char_tokenizer.py` — 소스 문자 vocab 구축/인코딩(단어→문자ids + 경계)
2. `src/char_encoder.py` — CharCNN 합성기 + MCE 인코더, 단위 테스트(shape/마스킹)
3. `src/model.py` — `encoder_type="bpe"|"mce"` 플래그로 MCE 추가(기존 경로 보존)
4. 데이터/collate — 소스(단어×문자) + 타깃(BPE) 배치
5. 토이 overfit으로 학습 동작 검증
6. OPUS 본 학습 → 베이스라인과 동일 데이터·스텝·평가로 BLEU 정면 비교
7. **전문어 테스트셋**(말뭉치/벤치마크류 50~100개) 별도 측정 → 가설 검증, (선택) 학술 파인튜닝 비교

## 평가 (성공/실패 모두 결과물)

- **H1 (핵심)**: 저빈도 전문어 번역 정확도 — 전용 테스트셋 + 전체 BLEU
- **H2**: 인코더 시퀀스 길이/속도 (단어 단위라 BPE보다 짧을 것으로 기대)
- 비교는 베이스라인 대비 ablation. MCE가 BPE를 못 이겨도, "왜 BPE가 강한가"의 분석이 산출물.

## 리스크

- 문헌상 문자/형태소 인코더가 BPE를 못 이기는 경우가 흔함 — 결과를 장담하지 않는다.
- 단어 합성기가 단어마다 동작 → 학습이 다소 느려질 수 있음.
- 한국어 공백(어절)이 형태소 경계와 완전히 일치하지는 않음(작동에는 무방).
- 순수 PyTorch 구현이라 ROCm 커널 리스크는 없음.
